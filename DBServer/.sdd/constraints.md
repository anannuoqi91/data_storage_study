# 系统约束

## 安全性
| 约束ID | 规则 | 影响范围 |
|--------|------|---------|
| SEC-01 | 查询入口只允许固定查询能力，不对外开放任意写 SQL | 查询端 |
| SEC-02 | 删除操作必须先标记分区 deleting，再等待冲突查询退出 | 删除流程 |
| SEC-03 | 查询必须在开始时绑定固定 release，执行中不允许漂移到新 release | 查询端 |
| SEC-04 | 元数据和 release 切换操作必须串行进行 | 发布、封口、回滚、删除、重建 |

## 合规性
| 约束ID | 规则 | 影响范围 |
|--------|------|---------|
| CMP-01 | 所有物理分区和保留边界统一按 UTC 解释 | 分区、删数、查询 |
| CMP-02 | 删除不可回滚，保留边界只允许单向前进 | 删除流程 |
| CMP-03 | 同一个对外 release 里不允许混用多个 schema_version | release 管理 |

## 性能
| 约束ID | 规则 | 影响范围 |
|--------|------|---------|
| PRF-01 | 设备规模第一版只支持 4 台 | 接入、缓冲 |
| PRF-02 | 发布粒度固定为 10 分钟窗口 | 发布、rollup |
| PRF-03 | 查询应优先读取派生数据，减少明细文件全扫 | trace 查询、聚合查询 |

## 可靠性
| 约束ID | 规则 | 影响范围 |
|--------|------|---------|
| REL-01 | 不引入 WAL，允许崩溃时丢失最近 1~5 秒内存缓冲 | 接入链路 |
| REL-02 | 只有完成元数据提交并切换 current_release 的结果才算正式生效 | 发布、封口、删除、回滚 |
| REL-03 | staging/orphan 文件必须能在重启后被清理 | 恢复 |
| REL-04 | `trace_index` 和 `rollup` 是可重建派生数据，不是唯一真相 | 派生数据管理 |

## 业务规则
| 约束ID | 规则 | 影响范围 |
|--------|------|---------|
| BIZ-01 | 第一版只围绕 `box_info` 一张主表，不扩展为通用多表框架 | 全系统 |
| BIZ-02 | 允许有限晚到，晚到上限 2 分钟 | 接入、发布、封口 |
| BIZ-03 | 不支持 2 分钟窗口之外的历史补录和业务纠正 | 接入、回放 |
| BIZ-04 | `lane_id` 为空统一映射为 `255` | 入库、rollup |
